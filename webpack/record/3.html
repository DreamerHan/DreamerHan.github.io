# 关于 Loader

# Loader 是什么

Loader 的概念主要源自于 webpack。

可以说支持 webpack 打包非 JS 文件的特性就是基于 Loader 机制来实现的。

它可以被看成一个特殊的转换器，通过使用特定的插件（babel-loader、css-loader、sass-loader 等）将对应类型的文件进行转换，这些文件最终被转换成能够被 webpack 处理的模块（webpack 只能打包 js）。

比如：babel-loader 可以将 ES6 代码转换为向后兼容的 ES5 代码，css-loader 可以将 CSS 文件转换为 JavaScript 模块，sass-loader 可以将 SCSS 文件转换为标准 CSS 文件等。

loader 还可以将文件从不同的语言（如 TypeScript）转换为 JavaScript 或将内联图像转换为 data URL。也是因为 loader 的存在才允许你直接在 JavaScript 模块中 import CSS 文件！

Loader 的代码本质

Loader 在代码层面，只是一个被暴露出来的 js 模块而已。而这个函数用于实现内容的转换，共有 3 个参数。

/**
 * @param {string|Buffer} [content] 源文件的内容
 * @param {object} [map] 可以被 https://github.com/mozilla/source-map 使用的 SourceMap 数据
 * @param {any} [meta] meta 数据，可以是任何内容
 */
function webpackLoader(content, map, meta) {
  // 你的webpack loader代码
}
module.exports = webpackLoader;

# Loader 的使用

# 两种方式

loader 有两种使用方式：

  • 配置方式 就是在 webpack.config.js 中的配置方式。
  • 內联方式 在每一个，每一个 import 语句中都显式的指定要使用的 loader。

# 配置方式

配置方式的语法关键字是 test、use 和 exclude 等等。

  • 通过 test 对应的正则去匹配要处理的文件
  • 通过 use 去指定要使用的 loader 和 要配置的细节
  • 通过 exclude 去指定不匹配的文件

同一个 rules 下，配置后的 loader 执行顺序是 从右到左，从下到上 的倒序模式。

像下面这种配置，会先执行 sass-loader 将 sass 文件转换成标准的 css 文件。然后 css-loader 接收到 sass-loader 转换的结果后将其变成 js 模块。最后通过 style-loader 把 css 放到 style 标签中同时写入 html 文件。

module.exports = {
  module: {
    rules: [
      {
        test: /\.css$/,
        use: [
          { loader: "style-loader" },
          {
            loader: "css-loader",
            options: {
              modules: true,
            },
          },
          { loader: "sass-loader" },
        ],
      },
    ],
  },
};

# 內联模式

内联模式使用 ! 将资源中的 loader 分开。每个部分都会相对于当前目录解析。

可以通过 ? 来传递查询参数，如 ?key=value 或者 ?{"key": "value", "foo": "bar"}

import Styles from "style-loader!css-loader?modules!./styles.css";

如果内联语句前面加入了前缀 ! 或者 !! 或者 -!，会覆盖配置中所有的 loader，preLoader 和 postLoader

  • 使用 ! 前缀，将禁用所有已配置的 normal loader(普通 loader)
import Styles from "!style-loader!css-loader?modules!./styles.css";
  • 使用 !! 前缀，将禁用所有已配置的 loader（preLoader, loader, postLoader）
import Styles from "!!style-loader!css-loader?modules!./styles.css";
  • 使用 -! 前缀，将禁用所有已配置的 preLoader 和 loader，但是不禁用 postLoaders
import Styles from "-!style-loader!css-loader?modules!./styles.css";

# 小结一下

  1. Loader 支持链式调用。

    一组链式的 loader 将按照倒序的模式执行。

    链中的每个 loader 会将转换应用在已处理过的资源上。链中的第一个 loader 将其结果（也就是应用过转换后的资源）传递给下一个 loader，依此类推。

    最后，链中的最后一个 loader，返回 webpack 所期望的 JavaScript。

  2. Loader 的引用与配置

  • 在 module.rules 中使用 loader 字段指定一个 loader 名称，相当于直接引用了这个 loader 模块。

    所以在解决 react 报错时，通过 loader 指定 babel-loader 时，如果没有安装就会报错。也不需要在文件顶部 import babel-loader。

  • Loader 可以通过 options 对象配置（仍然支持使用 query 参数来设置选项，但是这种方式已被废弃）。

  1. 其他

  • Loader 可以是同步的，也可以是异步的。它运行在 Node.js 中，并且能够执行任何操作。

  • 可以通过 package.json 的 main 来将一个 npm 模块导出为 loader

# 在项目里测试自己的 loader

  1. 在项目创建一个 loaders 文件夹，来管理所有的自定义 loader。

loaders/test-loader.js

// map 和 meta 没有尝试过
function testLoader(content, map, meta) {
  console.log("我是一个测试的loader");
  console.log("content", content);
  return content;
}

module.exports = testLoader;
  1. 调整 webpack.config.js 配置
  • module.rules 中配置自己的 loader
  • resolveLoader 来配置 loader 的获取来源有哪些
module.exports = {
  // 上面的代码

  module: {
    rules: [
      {
        test: /\.js$/,
        exclude: /node_modules/,
        use: [
          {
            loader: "babel-loader",
            options: {
              presets: ["@babel/preset-env", "@babel/preset-react"],
            },
          },
          {
            loader: "test-loader", // 这里设置了自己的 loader
          },
        ],
      },
    ],
  },

  // 只是单独设置 loader，会从 node_modules 中寻找对应的 loader 文件。自定义的loader就会报错
  // 通过 resolveLoader.modules 来配置项目的 loader 来源
  resolveLoader: {
    modules: [
      path.resolve(__dirname, "node_modules"),
      path.resolve(__dirname, "loaders"),
    ],
  },

  // plugins代码
};
